# 2026-07-07 배포 전략: Rolling, Blue/Green, Canary, Feature Flag

## 목표

배포 전략을 단순히 이름만 외우지 않고, 실제 운영에서 어떤 기준으로 선택하는지 정리한다.

배포 전략을 비교할 때는 아래 기준으로 봐야 한다.

```text
서비스 중단 시간이 있는가?
롤백이 빠른가?
리소스를 얼마나 더 써야 하는가?
새 버전과 기존 버전이 동시에 떠 있어도 되는가?
DB 스키마 변경과 호환되는가?
문제를 얼마나 작은 범위에서 발견할 수 있는가?
트래픽 제어 기능이 필요한가?
모니터링과 자동 롤백이 필요한가?
```

## 1. Recreate Deployment

## 개념

Recreate는 가장 단순한 배포 방식이다.

기존 버전을 내리고 새 버전을 올린다.

```text
v1 종료
→ v2 시작
```

예:

```bash
docker stop app
docker rm app
docker run -d --name app -p 8080:8080 app:v2
```

## 장점

```text
구현이 가장 쉽다.
추가 인프라가 거의 필요 없다.
서버 한 대에서도 바로 가능하다.
배포 흐름이 단순해서 이해하기 쉽다.
```

## 단점

```text
배포 중 서비스 중단이 발생한다.
새 컨테이너가 늦게 뜨면 중단 시간이 길어진다.
새 버전이 실패하면 복구 전까지 장애가 이어진다.
롤백도 다시 기존 버전을 내려받고 실행해야 해서 느릴 수 있다.
```

## 리소스 관점

```text
추가 서버 필요 없음
추가 컨테이너 필요 없음
비용은 가장 낮음
대신 가용성이 가장 낮음
```

## 적합한 상황

```text
개인 프로젝트
트래픽이 거의 없는 서비스
잠깐 중단되어도 되는 내부 도구
초기 실습 환경
```

## 부적합한 상황

```text
사용자가 계속 접속하는 운영 서비스
결제, 인증, 주문처럼 중단 영향이 큰 서비스
배포 실패 시 빠른 복구가 필요한 서비스
```

## 한 줄 정리

```text
Recreate는 가장 단순하고 저렴하지만 배포 중 중단이 발생하는 전략이다.
```

## 2. Rolling Deployment

## 개념

Rolling Deployment는 여러 인스턴스나 Pod를 조금씩 새 버전으로 교체하는 방식이다.

예:

```text
v1 v1 v1 v1
→ v2 v1 v1 v1
→ v2 v2 v1 v1
→ v2 v2 v2 v1
→ v2 v2 v2 v2
```

Kubernetes Deployment의 기본 전략이 RollingUpdate다.

## 장점

```text
무중단 배포가 가능하다.
리소스를 한 번에 2배로 늘리지 않아도 된다.
Kubernetes, ECS 같은 플랫폼에서 기본적으로 지원한다.
점진적으로 교체하므로 급격한 트래픽 전환이 없다.
```

## 단점

```text
배포 중 v1과 v2가 동시에 존재한다.
v1과 v2가 같은 DB 스키마를 함께 사용할 수 있어야 한다.
문제가 있는 버전이 일부 사용자에게 노출될 수 있다.
롤백 중에도 다시 점진적으로 교체해야 해서 즉시 전환보다 느릴 수 있다.
```

## 리소스 관점

Rolling은 설정에 따라 추가 리소스가 조금 필요할 수 있다.

Kubernetes 예:

```text
replicas: 4
maxSurge: 1
maxUnavailable: 0
```

의미:

```text
기존 4개를 유지하면서 새 Pod를 1개 더 띄울 수 있다.
배포 중 총 5개까지 늘어날 수 있다.
사용 불가능한 Pod는 0개로 유지한다.
```

리소스 특징:

```text
Blue/Green보다 추가 리소스가 적게 든다.
maxSurge 설정에 따라 일시적으로 리소스가 증가한다.
배포 속도와 안정성 사이에 트레이드오프가 있다.
```

## 트레이드오프

```text
maxSurge를 크게 하면 배포가 빠르지만 리소스를 더 쓴다.
maxUnavailable을 크게 하면 배포가 빠르지만 가용성이 떨어질 수 있다.
maxUnavailable을 0으로 두면 안전하지만 리소스 여유가 필요하다.
```

## 적합한 상황

```text
Kubernetes나 ECS 같은 오케스트레이션 환경
여러 인스턴스로 운영하는 일반적인 웹 서비스
대부분의 무중단 배포 기본값
```

## 부적합한 상황

```text
v1과 v2가 동시에 떠 있으면 안 되는 변경
DB 스키마가 하위 호환되지 않는 변경
새 버전을 미리 완전히 검증한 뒤 한 번에 전환해야 하는 경우
```

## 한 줄 정리

```text
Rolling은 가장 흔한 무중단 배포 전략이지만, v1/v2 동시 운영 호환성이 중요하다.
```

## 3. Blue/Green Deployment

## 개념

Blue/Green은 두 환경을 나눠두고, 현재 운영 환경과 새 버전 환경을 분리하는 방식이다.

```text
Blue = 현재 운영 버전
Green = 새 버전
```

전환 전:

```text
사용자 → Blue
Green은 대기
```

전환 후:

```text
사용자 → Green
Blue는 롤백용으로 대기
```

단일 EC2에서도 비슷한 구조를 만들 수 있다.

```text
Nginx
→ blue container:8080
→ green container:8081
```

운영 중에는 하나만 active로 둔다.

```text
사용자 → Nginx → blue
```

전환 후:

```text
사용자 → Nginx → green
```

## 장점

```text
새 버전을 미리 띄우고 health check할 수 있다.
전환이 빠르다.
문제가 있으면 이전 환경으로 빠르게 롤백할 수 있다.
배포 시점에 기존 버전을 바로 죽이지 않아도 된다.
```

## 단점

```text
환경을 두 벌 유지해야 하므로 리소스 비용이 크다.
전환 순간 트래픽이 한 번에 새 버전으로 이동한다.
DB 스키마 변경이 하위 호환되지 않으면 롤백이 어렵다.
세션, 캐시, 파일 저장 위치가 환경 간에 분리되어 있으면 문제가 생길 수 있다.
```

## 리소스 관점

Blue/Green은 리소스를 거의 2배 가까이 쓸 수 있다.

예:

```text
Blue: EC2 4대
Green: EC2 4대
```

또는 단일 EC2 blue/green:

```text
Blue container 1개
Green container 1개
```

단일 EC2에서는 비용 증가는 작지만, CPU/메모리는 배포 중 두 컨테이너를 동시에 감당해야 한다.

```text
CPU 여유 부족
메모리 부족
컨테이너 시작 실패
일시적 성능 저하
```

## 트레이드오프

```text
롤백은 빠르지만 리소스 비용이 크다.
새 버전 검증은 쉽지만 트래픽 전환은 한 번에 일어난다.
단일 인스턴스에서도 구현 가능하지만 EC2 자체 장애에는 취약하다.
```

## 적합한 상황

```text
빠른 롤백이 중요한 서비스
새 버전을 미리 띄우고 검증하고 싶은 경우
배포 중 중단을 줄이고 싶은 단일 서버 환경
인프라 리소스 여유가 있는 환경
```

## 부적합한 상황

```text
리소스 비용을 거의 2배 감당하기 어려운 경우
v1/v2가 같은 DB를 공유할 때 스키마 호환성이 없는 경우
트래픽을 조금씩 검증하고 싶은 경우
```

## 한 줄 정리

```text
Blue/Green은 빠른 전환과 롤백에 강하지만 리소스 비용과 DB 호환성 문제가 있다.
```

## 4. Canary Deployment

## 개념

Canary는 새 버전에 일부 트래픽만 먼저 보내고, 문제가 없으면 점점 늘리는 방식이다.

예:

```text
v1 99% / v2 1%
v1 95% / v2 5%
v1 90% / v2 10%
v1 50% / v2 50%
v1 0%  / v2 100%
```

목적은 새 버전의 위험을 작은 범위에서 먼저 검증하는 것이다.

## 장점

```text
문제가 있어도 영향 범위가 작다.
실제 사용자 트래픽으로 새 버전을 검증할 수 있다.
메트릭을 보면서 점진적으로 확대할 수 있다.
자동 롤백과 잘 어울린다.
대규모 서비스에서 안정성을 높일 수 있다.
```

## 단점

```text
구성이 복잡하다.
트래픽을 비율로 나눌 수 있는 인프라가 필요하다.
모니터링과 알람이 부족하면 효과가 떨어진다.
v1과 v2가 동시에 운영되므로 호환성이 필요하다.
DB 변경이 어렵다.
사용자별 일관성을 유지하려면 sticky routing이나 feature flag가 필요할 수 있다.
```

## 리소스 관점

Canary는 새 버전을 일부만 띄우므로 Blue/Green보다 리소스를 적게 쓸 수 있다.

예:

```text
v1 Pod 9개
v2 Pod 1개
```

또는:

```text
v1 95%
v2 5%
```

하지만 배포 중에는 v1과 v2가 동시에 떠 있어야 하므로 추가 리소스는 필요하다.

리소스 특징:

```text
Blue/Green보다 효율적일 수 있다.
Rolling보다 트래픽 제어가 정교하다.
관측성과 라우팅 인프라 비용이 더 들어갈 수 있다.
```

## 트레이드오프

```text
위험은 작게 시작할 수 있지만 운영 복잡도가 높다.
리소스는 Blue/Green보다 적을 수 있지만 모니터링과 자동화가 필요하다.
트래픽 분할은 강력하지만 세션, 캐시, DB 호환성 문제가 생길 수 있다.
```

## 카나리에 필요한 조건

```text
트래픽을 1%, 5%, 10%처럼 나눌 수 있어야 한다.
새 버전과 기존 버전이 동시에 떠 있어야 한다.
에러율, 응답 시간, CPU, 메모리, 비즈니스 지표를 볼 수 있어야 한다.
문제 발생 시 자동 또는 빠른 수동 롤백이 가능해야 한다.
DB 변경이 v1/v2 모두와 호환되어야 한다.
```

## 구현 방식

```text
Kubernetes + Ingress weight
Service Mesh traffic split
AWS ALB weighted target group
Feature Flag 기반 사용자 분할
API Gateway 기반 라우팅
```

## 적합한 상황

```text
트래픽이 큰 서비스
장애 영향 범위를 작게 시작하고 싶은 서비스
모니터링과 알람이 잘 갖춰진 팀
새 기능 안정성을 실제 트래픽으로 검증하고 싶은 경우
```

## 부적합한 상황

```text
모니터링이 부족한 서비스
트래픽 분할 인프라가 없는 서비스
v1/v2 동시 운영이 불가능한 변경
DB 마이그레이션이 하위 호환되지 않는 경우
```

## 한 줄 정리

```text
Canary는 가장 안전하고 정교한 점진 배포 전략이지만, 관측성, 자동화, 호환성이 갖춰져야 제대로 쓸 수 있다.
```

## 5. A/B Testing

## 개념

A/B Testing은 새 버전의 안정성보다 사용자 반응이나 비즈니스 지표를 비교하기 위한 전략이다.

Canary와 비슷하게 일부 사용자에게 다른 버전을 보여줄 수 있지만 목적이 다르다.

```text
Canary = 안정성 검증
A/B Testing = 실험과 지표 비교
```

예:

```text
A 그룹 → 기존 결제 화면
B 그룹 → 새 결제 화면
```

비교 지표:

```text
클릭률
가입률
구매 전환율
이탈률
체류 시간
```

## 장점

```text
사용자 반응을 데이터로 비교할 수 있다.
기능 변경의 비즈니스 효과를 검증할 수 있다.
Feature Flag와 잘 어울린다.
```

## 단점

```text
배포 안정성 검증과 목적이 다르다.
통계적으로 의미 있는 트래픽이 필요하다.
사용자 그룹 분리가 필요하다.
기능별 데이터 분석 체계가 필요하다.
```

## 리소스 관점

```text
두 버전의 화면이나 기능을 동시에 유지해야 한다.
서버 리소스보다 코드 복잡도와 분석 비용이 더 커질 수 있다.
Feature Flag 시스템이나 실험 플랫폼이 필요할 수 있다.
```

## 한 줄 정리

```text
A/B Testing은 배포 안전성보다 제품 실험과 사용자 반응 검증에 초점이 있는 전략이다.
```

## 6. Shadow Traffic

## 개념

Shadow Traffic은 실제 사용자 요청을 새 버전에도 복사해서 보내지만, 새 버전의 응답은 사용자에게 돌려주지 않는 방식이다.

```text
사용자 요청
→ v1이 실제 응답
→ v2에도 같은 요청 복사
→ v2 응답은 버림
```

목적은 새 버전이 실제 트래픽을 견딜 수 있는지 확인하는 것이다.

## 장점

```text
사용자에게 영향 없이 새 버전을 실제 트래픽으로 검증할 수 있다.
성능, 에러율, 로그를 비교할 수 있다.
대규모 변경 전 안정성 검증에 좋다.
```

## 단점

```text
구현이 복잡하다.
쓰기 요청을 복사하면 데이터가 중복 변경될 수 있다.
외부 API 호출이나 결제 요청이 중복되면 위험하다.
응답을 사용자에게 쓰지 않으므로 실제 사용자 경험 검증은 제한적이다.
```

## 리소스 관점

Shadow Traffic은 리소스를 많이 쓸 수 있다.

예:

```text
v1이 실제 트래픽 100% 처리
v2도 복사된 트래픽 100% 처리
```

이 경우 서버 처리량이 사실상 2배 필요할 수 있다.

## 트레이드오프

```text
사용자 영향은 거의 없지만 리소스 비용이 크다.
실제 트래픽 검증은 가능하지만 쓰기 작업은 조심해야 한다.
운영 복잡도가 높다.
```

## 한 줄 정리

```text
Shadow Traffic은 사용자 영향 없이 실제 트래픽으로 새 버전을 검증하지만, 리소스와 구현 복잡도가 크다.
```

## 7. Feature Flag

## 개념

Feature Flag는 배포와 기능 공개를 분리하는 방식이다.

코드는 이미 배포되어 있어도 기능은 꺼둘 수 있다.

```text
코드 배포 완료
기능 flag OFF
```

필요할 때 특정 사용자나 비율에만 기능을 켠다.

```text
내부 직원만 ON
관리자만 ON
5% 사용자만 ON
전체 사용자 ON
```

## 장점

```text
배포 없이 기능을 켜고 끌 수 있다.
문제가 생기면 빠르게 기능만 끌 수 있다.
Canary, A/B Testing과 잘 어울린다.
사용자 그룹별 기능 공개가 가능하다.
```

## 단점

```text
flag가 많아지면 관리가 복잡하다.
오래된 flag가 기술 부채가 된다.
조건 분기가 많아져 코드가 복잡해질 수 있다.
flag 상태에 따른 테스트 조합이 늘어난다.
```

## 리소스 관점

Feature Flag 자체는 서버 리소스를 크게 늘리지 않을 수 있다.

하지만 운영 비용은 다른 형태로 생긴다.

```text
flag 관리 시스템
flag 상태 저장
실험 그룹 관리
테스트 복잡도 증가
오래된 flag 제거 비용
```

## 트레이드오프

```text
기능 공개와 롤백은 쉬워지지만 코드와 운영 복잡도가 증가한다.
잘 관리하면 매우 강력하지만, 방치하면 기술 부채가 된다.
```

## 한 줄 정리

```text
Feature Flag는 배포와 릴리즈를 분리해 빠른 기능 제어를 가능하게 하지만, flag 관리 체계가 필요하다.
```

## 9. DB 마이그레이션 트레이드오프

배포 전략에서 가장 자주 놓치는 부분은 DB 호환성이다.

애플리케이션만 v1/v2로 나뉘는 것이 아니라, 두 버전이 같은 DB를 바라보는 경우가 많다.

위험한 변경:

```text
컬럼 삭제
컬럼 이름 변경
테이블 삭제
기존 enum 값 제거
필수 컬럼 추가
응답 스키마 비호환 변경
```

Rolling, Canary, Blue/Green 모두 v1과 v2가 동시에 존재할 수 있으므로 DB 변경은 하위 호환 방식으로 해야 한다.

안전한 흐름:

```text
1. 새 컬럼 추가
2. v1과 v2 모두 동작 가능하게 코드 배포
3. 데이터 백필
4. 새 코드가 새 컬럼 사용
5. 충분히 안정화 후 기존 컬럼 제거
```

한 번에 하면 위험한 흐름:

```text
컬럼 이름 변경
→ v2 배포
→ v1이 아직 남아 있음
→ v1이 기존 컬럼을 찾지 못해 장애
```

## 10. 실무에서 많이 쓰는 조합

## 작은 서비스

```text
Recreate
또는 단일 EC2 Blue/Green
```

이유:

```text
구현이 쉽다.
리소스가 적게 든다.
트래픽이 크지 않으면 충분할 수 있다.
```

## 일반적인 Kubernetes 서비스

```text
Rolling Deployment
```

이유:

```text
Kubernetes 기본 전략이다.
무중단 배포가 가능하다.
추가 리소스가 Blue/Green보다 적다.
```

## 안정성이 중요한 서비스

```text
Canary
+ 자동 롤백
+ 모니터링
```

이유:

```text
장애 영향을 작은 비율로 제한할 수 있다.
실제 트래픽으로 검증할 수 있다.
```

## 제품 실험이 많은 서비스

```text
Feature Flag
+ A/B Testing
```

이유:

```text
배포와 기능 공개를 분리할 수 있다.
사용자 그룹별 실험이 가능하다.
```

## 큰 변경 또는 성능 검증

```text
Shadow Traffic
+ Canary
```

이유:

```text
먼저 사용자 영향 없이 실제 트래픽으로 검증한다.
이후 일부 사용자에게만 점진적으로 공개한다.
```

## 11. 현재 단일 EC2 blue/green 구조의 위치

현재 이야기한 구조는 아래에 가깝다.

```text
단일 인스턴스 Blue/Green
+ Nginx reverse proxy
+ Docker container switching
```

구조:

```text
EC2
├── Nginx: 80/443
├── blue container: host 8080
└── green container: host 8081
```

전환:

```text
Nginx proxy_pass 8080
→ Nginx proxy_pass 8081
```

이 방식은 로드밸런싱이라기보다 트래픽 스위칭이다.

```text
blue 100% / green 0%
→ blue 0% / green 100%
```

장점:

```text
단일 컨테이너 교체보다 중단 시간이 적다.
새 컨테이너 health check 후 전환할 수 있다.
문제 발생 시 이전 포트로 빠르게 되돌릴 수 있다.
구현 난이도가 Kubernetes Canary보다 낮다.
```

한계:

```text
EC2 자체가 죽으면 전체 장애다.
진짜 고가용성은 아니다.
CPU/메모리가 두 컨테이너를 동시에 감당해야 한다.
트래픽을 5%, 10%처럼 정교하게 나누는 구조는 아니다.
DB 마이그레이션 호환성 문제는 그대로 남는다.
```

## 12. 어떤 전략을 선택할 것인가?

## 선택 기준

```text
서버가 몇 대인가?
트래픽이 얼마나 큰가?
중단을 허용할 수 있는가?
롤백이 얼마나 빨라야 하는가?
리소스 비용을 얼마나 감당할 수 있는가?
모니터링이 충분한가?
트래픽을 비율로 나눌 수 있는가?
DB 변경이 하위 호환되는가?
팀이 운영 복잡도를 감당할 수 있는가?
```

## 추천 판단

```text
단일 서버이고 중단만 줄이고 싶다
→ Blue/Green container switching

Kubernetes 기본 운영이다
→ Rolling Deployment

새 버전을 일부 사용자에게만 먼저 검증하고 싶다
→ Canary Deployment

기능 공개와 배포를 분리하고 싶다
→ Feature Flag

사용자 영향 없이 실제 트래픽을 새 버전에 태워보고 싶다
→ Shadow Traffic

제품 지표를 비교하고 싶다
→ A/B Testing
```

## 최종 정리

```text
Recreate는 단순하지만 중단이 있다.
Rolling은 가장 흔한 무중단 배포 기본 전략이다.
Blue/Green은 빠른 전환과 롤백에 좋지만 리소스 비용이 크다.
Canary는 가장 정교한 점진 배포 전략이지만 관측성과 자동화가 필요하다.
A/B Testing은 안정성보다 제품 실험에 가깝다.
Shadow Traffic은 사용자 영향 없이 실제 트래픽 검증이 가능하지만 비용과 복잡도가 높다.
Feature Flag는 배포와 기능 공개를 분리하지만 관리하지 않으면 기술 부채가 된다.
```

## 한 줄 결론

```text
실무에서는 Rolling을 기본으로 많이 쓰고, 안정성이 중요한 서비스는 Canary를 붙이며, 빠른 전환과 롤백이 필요한 경우 Blue/Green을 사용하고, 기능 공개 제어는 Feature Flag로 분리하는 경우가 많다.
```
